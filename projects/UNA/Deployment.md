# Phase 1 deployment from scratch

## 1) Clean and prepare the project locally

Before touching Vercel:

1. Make sure the app runs locally without errors.
2. Commit all code to Git.
3. Add `.env`, `.env.local`, and similar secret files to `.gitignore` if they are not already ignored.
4. Rotate the exposed secrets and update your local env file with the new values. ([Better Auth][2])

For your app, the minimum env set is likely:

- `DATABASE_URL`
- `BETTER_AUTH_SECRET`
- `BETTER_AUTH_URL`
- `UPLOADTHING_TOKEN`

From your previous files, those are the core ones you are already using.

---

## 2) Create a dedicated GitHub repository if you have not already

Vercel’s normal workflow is Git-based deployment. It supports GitHub, GitLab, Bitbucket, and Azure DevOps, and automatically deploys on pushes. ([Vercel][1])

Recommended structure for this phase:

- `main` branch = stable feedback version
- optional feature branches = work in progress

For your testing flow, keep `main` as the client-facing demo branch.

---

## 3) Create a fresh Vercel account or use your existing free account

If you are only collecting feedback for now, using Hobby temporarily is fine.

Inside Vercel:

1. Sign in
2. Click **Add New...**
3. Choose **Project**
4. Import your GitHub repository

Vercel will detect Next.js automatically for your project type. Vercel’s Next.js framework support is native. ([Vercel][3])

---

## 4) Configure the project during import

When Vercel shows the import screen:

- Framework preset: **Next.js**
- Root directory: leave default unless your app is in a subfolder
- Build command: usually leave default
- Output directory: leave default
- Install command: leave default unless you have a custom package manager setup

Because your app is a standard Next.js App Router project, Vercel should not need custom build configuration for the first deploy. ([Vercel][3])

---

## 5) Add environment variables in Vercel before the first deploy

In the import flow, or later in **Project Settings → Environment Variables**, add your variables.

For Phase 1, I recommend adding them to:

- **Production**
- **Preview**
- optionally **Development**

Vercel supports environment variables scoped by environment, and preview variables apply to non-production branches. ([Vercel][4])

### Add these first

- `DATABASE_URL`
- `BETTER_AUTH_SECRET`
- `BETTER_AUTH_URL`
- `UPLOADTHING_TOKEN`

### Set `BETTER_AUTH_URL` carefully

For this temporary testing phase, set `BETTER_AUTH_URL` to your Vercel project URL after the first deployment, for example:

```bash
BETTER_AUTH_URL=https://your-project-name.vercel.app
```

Better Auth documents `BETTER_AUTH_SECRET` as required, and your deployment URL should match the actual deployed app URL once live. ([Better Auth][2])

### Important note on `NEXT_PUBLIC_*`

Only variables prefixed with `NEXT_PUBLIC_` are exposed to the browser in Next.js. Server-only secrets should stay unprefixed. ([Next.js][5])

---

## 6) Deploy once to get the Vercel URL

Click **Deploy**.

Vercel will build the project and give you a URL like:

```text
https://your-project-name.vercel.app
```

This is the URL you will use for client feedback in Phase 1.

If the build succeeds, open the site and note the exact URL because you may need it for auth-related settings like `BETTER_AUTH_URL`.

## 7) Update `BETTER_AUTH_URL` if needed and redeploy

If you deployed before setting the exact final temporary URL, go back to:

**Project Settings → Environment Variables**

Update:

```bash
BETTER_AUTH_URL=https://your-project-name.vercel.app
```

Then redeploy.

This is important because auth flows often depend on the canonical deployed base URL. Better Auth’s docs call out `BETTER_AUTH_SECRET`, and its Next.js integration assumes a proper deployment URL setup. ([Better Auth][2])

Because your app uses Prisma + Neon, the production-like Vercel deployment needs a working database schema.

From your local machine, against the Neon database you want this testing app to use, run the schema sync steps you already documented for your project:

Your current project brief also says to restart the dev server after schema changes, but for deployment the important part is making sure the remote DB schema matches the code before or during the first real test.

That keeps feedback/demo content separate from the final launch data.

## 10) Use the right deployment type for feedback

This is a very important detail for your stack:

### Best choice for client feedback:

Use the **production deployment on `main`** with the autogenerated `*.vercel.app` URL.

### Avoid depending on preview branch URLs for upload testing

UploadThing’s docs warn that Vercel preview deployments are often protected by authentication by default, which can block UploadThing callbacks from reaching your app. ([docs.uploadthing.com][6])

So for your project:

- if you need uploads/auth/admin to work normally during feedback, share the **main deployment URL**
- do not rely on private/protected preview URLs for full end-to-end testing

That means your Phase 1 can still be on Hobby, but the best testing target is the main `.vercel.app` deployment, not an isolated feature preview URL.

## 11) If the deployment fails, check these places first

### Build logs in Vercel

Open the failed deployment and inspect:

- install logs
- build logs
- runtime logs

### Most common causes in your stack

- missing environment variables
- wrong `BETTER_AUTH_URL`
- Prisma schema/database mismatch
- stale database/auth state
- wrong UploadThing token
- uploaded image host not allowed in Next config

Vercel’s Git deployment flow gives you deployment logs for each push, which is the fastest way to identify failures. ([Vercel][1])
